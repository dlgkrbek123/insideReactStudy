CSR
- 클라이언트측에서 실행되는 react가 js를 통해 동적으로 DOM 트리 생성
- 단일 html이 응답됨 (+bundle.js)

- 서버를 운영해야할 필요가 없음 (cloudfront + s3)
- 한 번 번들이 로드되면 이후 페이지전환 및 UI 업데이트가 클라에서만 처리되므로 
  빠르다.

- html 응답되고 번들 다운받고 실행될 떄까지 빈화면을 본다
  - FCP가 늦다 (특히 번들이 클수록 심화됨)
  - 초기 로딩 속도가 느림
- 초기 HTML이 비어있어 SEO가 좋지 않다

- html 수신 - parseHTML - layout - paint => 빈화면
- 번들 다운로드 - 파싱 - 실행 - 동적 UI 생성(layout - paint)

------------------------------------------------------

SSR
- 초기 렌더링시 요청받은 페이지에 대한 동적으로 HTML을 생성해 브라우저로 전달 (+js 번들또한 같이 응답)
- 페이지 전환시에는 클라이언트 사이드 렌더링처럼 번들에 의해 다음 화면이 렌더링

- 개선된 FCP 
  - js번들 다운로드 및 실행을 기다리지 않고 거의 모든 콘텐츠를 바로 전달받으므로 로딩속도가 대개 개선됨
- TTFB는 좀 느려짐
- SEO가 개선됨
- 서버를 운영하고 관리해야함

- 초기 렌더링의 책임이 서버에 있다.
- 컴포넌트가 서버에서도 실행될 수 있고 클라에서도 실행된다. (코드 짤 떄 양쪽을 다 신경써야함)
- getServersideProps같이 서버 전용 함수의 코드는 번들에 미포함

Hydration
- SSR에서 전달된 정적인 HTML에 대해 js번들이 실행되어 상호작용이 가능한 상태로 활성화하는 것
- 이벤트 핸들러 부착, 동적인 상태관리가 가능해짐
- SSR된 HTML과 클라이언트가 생성된 가상 DOM 구조가 일치해야함

직접 서버사이드 렌더링 구현하려면 쉽지 않음 => Nextjs 쓰자...
스트리밍 : html을 조각내어 순차적으로 클라에 전송

renderToString()
- 리액트 컴포넌트 렌더링 결과를 HTML 문자열로 응답해주는 함수
- SSR의 근간 (의사코드로 구현한다고 한다면 아래처럼...)
  - 렌더링된 html과 initialProps을 리턴하게 구성해야함
  - hydrateRoot를 통해 hydration 적용
- 동기적으로 작동하여 SSR하는 동안 서버 메인 스레드가 묶임

renderToNodeStream()
- renderToString은 전체 HTML이 완료될떄까지 기다려야함 (+메모리)
- ReadableStream을 사용하여 렌더링된 HTML을 chunk로 클라에 전송
  - Suspense를 미지원
  - 따라서 TTFB는 빠르나 모든 데이터가 준비되어 HTML스트림이 완료될떄 까지 기다려야함

renderToPipeableStream()
- 렌더링이 준비된 부분부터 클라이언트로 전송
- 인터랙션이 있는곳부터 선택적인 수화과정 진행
- Suspense지원
  - SSR시 폴백 UI전달
  - 데이터 로딩 완료시 html을 재전달하여 교체
- Suspense 제외한 애들 렌더링 완료시 초기 UI준비가 끝나서 html 스트리밍 시작
- 폴백은 <template>으로 감싸져 있다.
  - 실제 콘텐츠가 여기에 삽입됨
  - UI 각 부분을 독립적인 chunk로 스트리밍 가능

초기 로딩 속도를 극대화 + 서버 자원을 보존

SSG (Static Site Generation)
- 빌드 시점에 html을 만들기
- html을 만드는 시점만 다르지 Hydration 동일
- getStaticProps()
  - 빌드할때 initialProps를 계산할떄 호출
- getStaticPaths()
  - 정적으로 생성할 페이지 경로를 정의
  - 미포함 경로는 SSR한번하고 캐싱됨

ISR (Increment Static Regeneration)
- 정해진 시간간격 이후 페이지 요청시 html재생성
- 다음 접속자 부터는 새로 만든 페이지 보게함
- revalidate

------------------------------------------------------

FCP(First Contentful Paint) : 화면에 의미있는 콘텐츠가 그려지는 시점 (시각적)
TT(Time to Interactive) : 페이지가 유저 입력에 실제로 상호작용이 가능해지는 시점
LCP(Largest Contentful Paint) : 뷰포트에서 가장 큰 컨텐츠가 렌더링되는 시간
